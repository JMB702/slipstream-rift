TL;DR — the three channels feeding an agent

Every NPC agent receives information through exactly three channels.

Everything else the agent "knows" must come from one of these. There is no fourth backdoor.

1. SYSTEM PROMPT static, authored in the ElevenLabs dashboard NPC persona only; no live data 2. KNOWLEDGE BASE 3 documents, attached at agent level Slipstream Backstory Slipstream Map: FPS Shooter Arena Slipstream Map: Original Arena Synced via pnpm sync:kb 3. CONTEXTUAL UPDATES one-shot system messages pushed via SDK a) ONE big memoryBlob at session start b) npc_alerts mid-session (8 kinds)

1. Chat histories

Storage

Every transcript ClientMessage hits acceptTranscript() at apps/party/src/server.ts:909, which delegates to GameStorage.appendTranscript() at apps/party/src/storage.ts:102.

PartyKit Durable Object key shape

tx:<npcId>:<playerName>  →  Array<{ role, text, at }>
  • Capped at 200 lines per pair — TRANSCRIPT_CAP at storage.ts:14. Older lines shift out FIFO.
  • Persists across rounds and PartyKit restarts — Durable Object storage lives in Cloudflare Workers KV.
  • Tied to playerName, not a UUID. Two players named "Bob" share state. (V1 known limitation.)

What reaches the agent

The blob is built by buildMemoryBlob() at apps/party/src/server.ts:1038. It's a Markdown string, hard-capped at 8192 characters at server.ts:1176.

The eavesdropping feature is real, not just docs The "last 5 lines with anyone in earshot recently" is implemented in code at server.ts:1046–1049 + storage.ts:134. NPCs really do see what they've recently said to other players when talking to you.

How it's delivered to ElevenLabs

The blob travels from PartyKit to the client to the SDK in three hops.

SERVER CLIENT ELEVENLABS SDK ───────────── ───────────────────────── ────────────────────── buildMemoryBlob() │ ▼ npc_context ───▶ manager.ts: on('npc_context') ServerMessage │ ▼ ConvAISession.start() ───▶ conversation ConvAISession.ts:124 .sendContextualUpdate(blob) │ ▼ Agent's NEXT turn gets blob as system msg
Critical mechanic — read this The blob is NOT a prompt override, NOT a dynamic variable, NOT a knowledge base entry. It's a one-shot system message via the SDK's sendContextualUpdate(). It's injected into the conversation transcript once, before the agent's first substantive turn. If the session is long, the blob will gradually fall out of the context window as the conversation grows past whatever the agent's context limit is.

Live alerts (the only mid-session signals)

The bridge is formatNpcAlert() in apps/party/src/server.ts and the dispatch helper dispatchNpcAlert() just below it. Every alert is pushed through sendContextualUpdate() on the ElevenLabs SDK (apps/client/src/voice/manager.ts:87–91). Eight alert kinds exist; the three combat alerts were original, the other five were added to close the gap.

kindFires whenWhat the agent learns
damaged This NPC was just hit Shooter name, damage amount, HP after, was-killed flag
shot_fired This NPC fired its own rifle Victim name (or intended), hit/miss, killed flag
friend_attacked One of this NPC's friends was just hit Friend name, attacker name
kill_witnessed A kill happens elsewhere on the map (NPC is not killer / victim / friend-of-victim) Killer name, victim name
hostility_ended This NPC's 30-second hostility timer toward someone just expired The name they're no longer angry at
npc_befriended_player A DIFFERENT NPC just crossed the friendship threshold with a player Which NPC, which player
player_reloaded The player this NPC is talking to started a reload Player name (~1.5s defenseless window)
player_joined A new human connected to the room Player name
player_left A human disconnected (and was NOT this NPC's current conversation partner) Player name

All alerts route through dispatchNpcAlert() which looks up the bot's botConversationWith connection and pushes a formatted system message. activeSessionBots() generator on the server iterates every NPC currently in a voice session.

Where action state lives

Persisted (PartyKit DO storage)

  • Chat transcripts — tx:<npcId>:<playerName>
  • Friendship score — friend:<npcId>:<playerName> (storage.ts:72)
  • Consent record — consent:<playerName>
  • Last-session timestamps — for the sense-of-time greeting buckets

Survives restarts. Survives rounds.

In-memory only (lost on restart)

  • Hostility entries — ServerPlayer.hostility[] (state.ts:33), pruned each tick
  • Follow target — botFollowing
  • Flee target + until — botFleeingFrom
  • Per-round kill/death counters
  • HP, ammo, position — game state, regenerates from inputs

If the PartyKit room restarts, all of this resets.

The shape of the gap

Today an agent's worldview is: persona + 3 lore docs + a session-start snapshot + 8 flavors of live alert covering combat, social shifts, hostility timer expiry, the partner's reload, and the room roster changing under them.

What an NPC now hears mid-session

  • It got shot (damaged)
  • It fired its own rifle (shot_fired)
  • A friend got shot (friend_attacked)
  • Someone else was killed across the map (kill_witnessed)
  • Its own hostility timer just ended (hostility_ended)
  • Another NPC just made a new friend (npc_befriended_player)
  • The player they're talking to is reloading (player_reloaded)
  • A human joined or left the room (player_joined / player_left)

Still silent (by design or by absence)

  • Shots from other NPCs that missed and hit nothing — too noisy
  • "Walked into the open," cover state, stance — no server-side model
  • Weapon swap — not implemented in the engine
  • Health regen ticks, ammo count changes on partner
  • Friendship score nudges that don't cross the threshold
  • Other NPCs' follow / flee decisions
Adding more alerts is mechanical Pattern: define a new variant of the NpcAlert union in apps/party/src/simulation.ts, add a formatter case in formatNpcAlert() in apps/party/src/server.ts, and emit the alert at the event source via this.dispatchNpcAlert() (looping this.activeSessionBots() when broadcasting to all live sessions). No new wire types, no schema changes — the npc_alert ServerMessage carries arbitrary text.
