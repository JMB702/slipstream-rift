B1 Voice session drops while player is stationary INSTRUMENTED PRIORITY: HIGH

Status — 5/16 Diagnostic telemetry shipped: voice_session_end ClientMessage now carries a reason field, the client tags drops as proximity | npc_disappeared | sdk_ended | sdk_error | teardown | manual, and the server emits a synthetic connection_closed end event for sessions in flight when the WebSocket drops. Next test session will distinguish "you walked away" from "the SDK quietly died" in the feedback report. Root-cause fix waits on first reproduction with the new data.

Symptom

During the Mira session, the voice connection ended and re-opened at least twice (~9:45:23 and ~9:55:25) while Jeff was standing still on a ledge. Each restart fired a fresh stock greeting mid-conversation despite full transcript continuity.

Jeff said "I wasn't walking at all. I was just standing on a ledge. I didn't get further away and then come closer. It just randomly dropped."

Why proximity hysteresis can't be the cause

The proximity loop tears down a session only when activeDist > NPC_VOICE.radius + NPC_VOICE.hysteresis (see apps/client/src/voice/manager.ts:148-157). A stationary player can't cross that threshold. So the disconnect originates elsewhere — likely the SDK or the agent side.

Hypotheses to investigate (in order)

  1. ElevenLabs idle/silence timeout. The agent may drop the session after N seconds of no user audio. Conversations had long thoughtful pauses. Check: ElevenLabs agent settings → conversation timeout / silence timeout.
  2. Cloudflared tunnel jitter. The webhook URL is on a free tunnel that occasionally renegotiates. If the SDK relies on a long-lived signaling connection that crosses the tunnel, a renegotiation could kill it. Check: cloudflared process logs around 9:45:23 and 9:55:25.
  3. SDK onError firing silently. The session-end branch in manager.ts doesn't distinguish error-end from normal end. Check: Browser console for SDK errors around those timestamps.
  4. Bot died and respawned mid-conversation. If Mira's bot took a hit during the conversation, the conversation might end on its death. Check: bot's alive/respawnAt in snapshots.

Action items

  • Add lifecycle telemetry to the capture layer. Emit a voice_session event with a reason field (normal | error | sdk_disconnect | bot_died | manual) so the report can group dropped sessions and we can see frequency.
  • Read the ElevenLabs agent's conversation timeout setting and bump it to something generous (e.g. 5 min of silence) if it's the cause.
  • Run cloudflared with --metrics and log restart events; cross-reference with disconnects.
  • Add SDK onError capture so the dropped sessions reveal an error message, not just a phase change.

B2 Greeting fires on session resume regardless of recency DONE PRIORITY: HIGH

Status — 5/16 Plumbing complete. GameStorage tracks last:<npcId>:<playerName> on every session end (including connection_closed). npc_context carries elapsedSinceLastMs. ConvAISession picks firstMessage by bucket: <90s → suppressed (no greeting), 90s-10m → resume line if persona has one, else suppressed, ≥10m → fresh greeting. memoryBlob gets a "## Time since you last talked" section so the LLM can phrase the gap naturally. Optional follow-up: author resumeLines per persona in the roster — until then, short-pause resumes are silent and the agent leans on the elapsed-time line in memoryBlob.

Symptom

When the voice session restarts (whether from B1 or from genuinely walking away and back), the new session opens with a stock greeting from npc.greetings. The memoryBlob push happens just after, so the agent does recall the prior conversation — but the agent's first audible line is still a fresh hello.

Mid-conversation injection examples 9:45:23 — "Mm — you. What's the situation."
9:55:25 — "Hey, hi, hey. Got a second? Or are you busy."

Jeff's framing

"Even if I had walked away from her and come back, it shouldn't start with a fresh greeting. Our characters probably need a better sense of time."

Design — "sense of time"

Greeting choice should be conditional on recency of the last session with this player:

Time since last sessionOpening behavior
< 90 secondsNo greeting. Open with "". Agent picks up from memoryBlob — "you wanted me to follow you, where to?"
90 sec – 10 minShort re-acknowledgment from a separate resume_lines pool. E.g. Mira: "Hey, back. You good?"; Guts: "Mm. Still here."
10 min – 1 dayTime-aware greeting acknowledging the gap. "Long time. What's the situation."
> 1 dayStandard npc.greetings pool, plus a contextual update with "Last time we talked: …"

Action items

  • Track lastSessionEndAt per (npcId, playerName) in GameStorage. Updated on voice_session_end. Cheap KV key: last:<npcId>:<playerName>.
  • Extend npc_context ServerMessage with elapsedSinceLastMs so the client can decide greeting behavior.
  • Add resumeLines field to NpcDef — short re-acknowledgments per persona. Wire roster.
  • Client picks firstMessage by recency bucket in ConvAISession.start. Empty string suppresses the greeting entirely.
  • Append "Time since we last talked: X minutes" to the memoryBlob in buildMemoryBlob so the LLM can reference it explicitly.

B6 Follow logic: behind-player + backward-walk + late-turn DONE PRIORITY: MED

Status — 5/16 (revised) Initial B6 fix dropped the "trail behind player's facing" computation (good). But the standoff-on-the-ray formulation still caused walks-backward-when-player-approaches: as the player closed in, the goal-on-the-ray slid past the bot's position toward the player, and the bot retreated to maintain 2.5m. Jeff caught this in the next test session.

Real fix: replaced the continuous-goal formulation with a discrete state machine (HOLD ↔ FOLLOWING) with hysteresis (inner 2.5m, outer 4m) so the player can walk past at close range without the bot retreating. On HOLD → FOLLOWING transitions the controller also sets botFollowHoldUntil = now + 500ms — during that window the movement section zeroes forward/right so the bot rotates toward the player FIRST, then walks. This subsumes B7's "yaw slew feels slow" — the bot is no longer trying to walk and turn at the same time.

Symptom

Whenever an NPC followed the player, the controller computed its goal as a point 3m off the player's facing direction (using player.yaw). Turning to face the NPC instantly flipped the "behind point" past them, so they'd walk around the player to get behind again — making it impossible to stand still and have a face-to-face conversation while followed.

Fix

Dropped the facing-direction math. New behavior: the goal is a point on the line from player → bot, BOT.followStandoffDist (2.5 m) away from the player. Player facing doesn't enter the calculation, so the bot stays wherever it already is when you turn around. If the bot is already inside the standoff radius, the goal is its own position — it stands still instead of micro-adjusting.

Files

  • packages/shared/src/constants.ts — new BOT.followStandoffDist = 2.5
  • apps/party/src/bots/controller.ts:207-225 — replaced trailing math with standoff-follow on player→bot line
Verify in-game Ask any NPC to follow you. Walk away in a straight line — they should keep pace behind you. Stop and turn 180°. Expected: they stay where they are (or finish closing into the 2.5m radius); they do NOT circle around to your new back. Walk sideways past them — they should re-close to standoff but not insist on a specific side.

B8 stop_following may not fire — tool failure vs. hallucinated refusal DONE PRIORITY: MED

Status — 5/16 Subsumed by B10. The session-after-B6 reproduction showed the pattern: zero tool_call events, but the regex fallback (applyTranscriptIntent) correctly set botFollowing server-side. The agent had no signal that state had changed and therefore claimed (falsely) that the system was refusing. B10 fixes that with self-state alerts on every state-change site.

Symptom

1:31:53 PMJEFF: All right, guts, you can stop following me now.
1:31:58 PMGUTS: Can't do that right now. System's not letting me. Still on your tail.

Two possibilities

  1. The agent actually called stop_following and got an error response (server returned 4xx) — and Guts's "System's not letting me" is the agent's in-character interpretation of the failure. This is concerning because we want a successful tool call here, not a graceful failure recap.
  2. The agent never called the tool and just hallucinated a technical excuse to stay in character. The persona may not weigh stop_following as something he'd readily do — but the description should be neutral enough that he uses it when asked directly.

Why we don't know yet

This session predated the F6 in-memory event ring buffer; the events log is empty for that window. We need to reproduce with the ring buffer hot.

Action items

  • Reproduce: follow with Guts (or any NPC), ask to stop. Run pnpm session:last immediately after.
  • If tool_call stop_following ok=true appears in events but the bot kept following → controller-side bug, not respecting bot.botFollowing = null.
  • If tool_call stop_following ok=false → look at server-side handler error (probably "player not in room" because the conn id changed).
  • If NO tool_call event → agent hallucinated. Tighten the tool description to make stop_following sound less like a "system override" and more like a routine action the agent willingly takes.

B10 Agent's body-state perception lags reality (broken feedback loop) DONE PRIORITY: HIGH

Symptom

Verbatim from pnpm session:last on the 5/16 4:12 PM Mira session (full transcript in the report; key moments below):

4:12:46JEFF: Hey, follow me, let's walk and talk.
4:12:46MIRA: Lead the way, I guess.
4:12:49MIRA: Huh, that's weird. It's not... it's not letting me. Like, I'm trying to move but my feet are stuck.
4:13:38JEFF: Whatever it's feeding you is not correct. This is a bug that we need to fix.

Root cause

Three independent code paths can change a bot's body state — and only one of them informed the agent:

  1. Tool handler (handleFollowTool et al.) — fired when the LLM successfully invoked the webhook. Emitted server tool_call events but no agent-visible signal.
  2. Regex fallback in applyTranscriptIntent — fired when the user's transcript matched "follow me" / "stop following" / etc. Mutated bot.botFollowing directly. No agent-visible signal.
  3. Damage cascade (markAttack) and natural expiry (pruneHostility) — change hostility lists. Some had partial alerts (friend_attacked, hostility_ended), some didn't.

In the captured session, Mira's body was following correctly (via the regex fallback). The LLM's internal model still said "not following" because nothing told it the state had changed, so it improvised a "system's not letting me" excuse — the same pattern as B8.

Fix

Closed the perception loop. New pushSelfStateAlert(bot, alert) helper on the server: if the bot has an active voice session, format an npc_alert system message and push it via the existing npc_alert ServerMessage. The SDK queues contextual updates for the agent's next turn, so the LLM sees the body-state confirmation before its next response.

Six new NpcAlert kinds added to simulation.ts:NpcAlert and server.ts:formatNpcAlert:

  • self_follow_started / self_follow_stopped (source: tool | regex | auto)
  • self_flee_started
  • self_attack_started / self_attack_stopped (source: tool | damaged | expire)
  • self_befriended_player

Wired into every state-change site:

  • handleFollowTool, handleFleeTool, handleStopAttackingTool, handleStartAttackingTool, handleMakeFriendTool
  • applyTranscriptIntent (the regex fallback that fired in the captured session)

The self_follow_started message text is deliberately direct:

[System: Your body just started following ${playerName}. You'll automatically
walk to stay near them — you don't need to do anything to keep up. If asked
whether you're following, say yes. If your feet "feel stuck," that perception
is wrong; trust this system message.]

Verify next session

  1. Walk up to Mira, say "follow me."
  2. Run pnpm session:last. Look for tool_call follow_player (or — if the agent didn't call the tool, the regex fallback path fired). Either way, the agent's next response should reference following correctly.
  3. Ask Mira "are you following me?" — she should say yes with conviction, not "my feet are stuck."
  4. Repeat with stop / flee / attack / friend tools.

Connection-closed observation (parked under B1)

The captured session's voice_session end event had reason=connection_closed after ~143s, which is the B1 disconnect pattern. Jeff didn't flag it this turn; logging it for cross-reference with B1.

B11 lean_wall / patrol / sprint_patrol verbal-only failures + missing telemetry DONE PRIORITY: MEDIUM

Symptom

Jeff said "leaning against the wall doesn't seem to work." pnpm session:last on the captured session surfaced two distinct failure modes for the three new movement tools (patrol, sprint_patrol, lean_wall):

  1. Vicky/fennel never called lean_wall — she verbally refused ("It seems I can't quite manage that right now") without invoking the webhook. Classic B8 pattern: agent hallucinates a refusal, server never sees the request.
  2. Mira DID call all three tools — every one returned npc not spawned. The events log showed tool_call patrol/sprint_patrol/lean_wall with ok:false and playerName='' (hardcoded empty in the emit). With playerName empty the lookup of the requesting human's snapshot fell through; the failure path also didn't surface which bots actually were in the room, so debugging from the report alone was impossible.

Root cause

  • Failure 1: The new tools had no regex fallback in applyTranscriptIntent. The follow / stop_following pair has had one since B8; the new tools were shipped without one, so any verbal-only refusal silently stranded the request.
  • Failure 2: Two compounding gaps. The handlers' failure-path emit calls passed playerName: '' (hardcoded) instead of the caller's name, so feedback reports couldn't tie failures to a player. And the npc_not_spawned branch returned a flat error with no inventory of the bots that were in the room — leaving the question "which npcId did the agent think it was?" unanswerable from logs.

Fix

Three changes in apps/party/src/server.ts:

  1. Factored shared helpers — applyPatrolToBot(bot, sprint) and applyLeanTargetToBot(bot) own the actual state mutations; tool handlers and the regex fallback both call them so the two entry points cannot drift.
  2. Telemetry repair — handlePatrolTool / handleLeanWallTool now take playerName as a parameter (route branches pass it through). Failure-path emits include args: { reason: 'npc_not_spawned', botsInRoom: [...] } via a new botInventory() helper. The 404 response body carries the same list so failed agent webhooks see the same diagnostics.
  3. Regex fallbacks added to applyTranscriptIntent for lean_wall, sprint_patrol, and patrol. Patterns cover the natural phrasings ("lean against the wall," "take cover behind the wall," "patrol the area," "run patrol," "stretch your legs"). Order matters: sprint_patrol is checked before patrol because "sprint patrol" contains "patrol." Fallback emits include args: { source: 'transcript' } so the feedback report can distinguish webhook calls from regex catches. Each successful fallback also pushes the matching self-state alert (self_lean_started, self_patrol_started) so the agent's perception stays aligned (B10 invariant).

Verify next session

  1. Walk up to an NPC; say "lean against the wall." Bot should walk to the nearest wall and adopt the lean pose.
  2. Same NPC, say "patrol the area." Bot resumes patrol. Then "sprint patrol" — bot patrols at sprint speed.
  3. Run pnpm session:last. Each command should appear as a tool_call event with either the webhook source (no source arg) or the regex source (source: 'transcript').
  4. If npc_not_spawned ever fires, the event's args now include botsInRoom — confirms the failure mode and tells you which NPCs the server actually has.

F5 NPC self-state / persona deltas DONE PRIORITY: HIGH

Why this matters NPC personas are baked into ElevenLabs agent system prompts — static. When in-fiction events change a character ("your shoulder pain is gone", "your friend Halsey is back"), we need a way to override specific facts without re-PATCHing the agent every time. F5 is the primitive that makes that possible. F1 (Halsey) will use the same plumbing to introduce "Halsey is alive" as a Mira delta when she walks into the arena the first time.

What was built

  • New DO storage key state:<npcId> holding a list of NpcStateEntry { at, summary, evidence?, source }. Capped at 50 entries.
  • GameStorage.getNpcState / appendNpcState / clearNpcState in apps/party/src/storage.ts.
  • buildMemoryBlob front-loads a new section: "## What's changed about you (authoritative — overrides your persona)". Each entry rendered with elapsed-since-changed. Layers above the world description so the LLM weighs it highest.
  • Admin route POST/GET/DELETE /admin/npc-state, secret-gated by the same ELEVENLABS_AGENT_TOOL_SECRET as the tool webhooks. No consent check (operator action, not player).
  • CLI: pnpm npc:state <npcId> "<summary>" [--evidence="…"] [--source=X]. Also --list <npcId> and --clear <npcId>.

Applied so far

Mira — shoulder pain healed Backfilled 5/16, 1:10 PM via pnpm npc:state mira "Your shoulder pain is gone…". Entry written with evidence pointing at the 5/16 9:35-9:56 AM transcript where Jeff committed. Next time you talk to Mira and bring up the shoulder, she should notice the absence in the moment — not preemptively announce it (the entry's summary explicitly instructs the LLM to discover, pause, react).

How to verify

  1. Run pnpm npc:state --list mira and confirm the entry round-trips from storage.
  2. Open a new voice session with Mira. Ask: "How's the shoulder?"
  3. Expected: she pauses, notices it doesn't hurt, references the conversation where you offered to fix it. Her jumpiness should remain unchanged (per the entry's wording).

Follow-up — in-fiction tool (deferred)

The script is for operator-side use. For future durable changes the agent itself should propose, add an accept_self_change webhook tool: when the player offers a change AND the agent agrees in character, the tool writes the delta. Same storage path, no operator step. Lower priority — the script-based flow is fine for now and prevents accidental persona drift from sloppy LLM tool calls.

F6 Durable workflow: admin HTTP + session CLI DONE PRIORITY: HIGH

Why this matters Until now, every time Jeff asked "look at my last session," I had to locate the right DO sqlite file, run sqlite3 with quoted hex, pipe through a node V8 deserializer, group sessions manually, and cross-reference the events JSONL — about 10 minutes of friction per session. F6 makes it one command. This is the path going forward for game-improvement feedback loops.

What was built

  • In-memory event ring buffer on the server (RECENT_EVENT_CAP = 1000) so events can be joined with transcripts via HTTP without relying on the tee'd JSONL file. Persists across onMessage handlers (module-level state is per-DO isolate); drops on DO restart, which is fine for operator telemetry.
  • Three new admin routes, secret-gated by ELEVENLABS_AGENT_TOOL_SECRET:
    • GET /admin/sessions?count=N&player=X&since=2h — decodes all transcripts in this room, groups into sessions by 5-min gap, returns last N enriched with overlapping events from the ring buffer and the NPC persona-delta state that was active at session start.
    • GET /admin/state — every NPC's persona-delta entries in one call.
    • GET /admin/snapshot — live game state in this room: who's playing, position, health, ammo, hostility (with ms remaining), follow targets, active conversation partners, friendsWith.
  • CLI wrapper pnpm session:last — queries both fps_shooter and arena rooms, merges, sorts by recency, pretty-prints as markdown. Flags: --player=X, --since=2h, --json, --no-content. Positional arg is session count.

The new workflow

  1. You play the game.
  2. When something interesting happens, you tell me about it.
  3. I run pnpm session:last (or pnpm session:last 3, or with a player filter). Single command, no SQLite, no V8 decoding.
  4. The output includes: transcript, tools fired during the window, hostility changes, shots, feedback signals, and the NPC state that was active. Everything I need to analyze + suggest fixes.

Known gap

Persona deltas are per-room, not per-NPC Mira's "shoulder pain is gone" delta was written into the fps_shooter DO. When Jeff played on arena next, the arena DO has no record of the delta — Mira there still has her pain. Same is true for transcripts and friendships (already documented in CLAUDE.md as a v1 limitation). The cleanest fix is a shared "global NPC state" key space outside any room's DO, or a startup hook that propagates deltas across rooms. Tracking as a follow-up; not in scope for F6.

Verify

  1. Server is hot-reloaded with the new routes (confirmed live by the smoke test that returned the 5/16 1:26 PM arena session).
  2. Run pnpm session:last — should print the most recent session as markdown.
  3. Run pnpm session:last 3 --no-content — three most recent across both rooms, summary only.
  4. Run pnpm session:last --json — raw structured data, pipeable to other tools.

F7 Game-change registry: NPCs notice every shipped change DONE PRIORITY: HIGH

Why this matters Hand-applying persona deltas with pnpm npc:state is for one-offs. Repeating the dance for every shipped mechanic (coffee, new characters, world-state shifts) × 7 NPCs × every map is fragile and easy to forget. F7 turns the persona-delta primitive (F5) into a repeatable workflow keyed off a single TypeScript registry.

What was built

  • Registry at packages/shared/src/game-changes.ts exports GAME_CHANGES: readonly GameChange[]. Each entry: { id, at, scope: 'all' | npcId[], summary, evidence? }.
  • Storage dedup: seeded:<id> boolean key in the DO. storage.isGameChangeSeeded / markGameChangeSeeded.
  • onStart hook: server.ts:seedGameChanges() walks GAME_CHANGES and writes every unseeded entry as a persona delta to every in-scope NPC's state:<npcId>, then marks the seed key. Logs [game-change] seeded <id> → N NPC(s) in room <map>.
  • First entry seeded: 2026-05-16-coffee-maker → 6 NPCs in fps_shooter (verified via pnpm npc:state --list mira; mira now has 2 entries).

The workflow going forward

  1. Ship the code change.
  2. Add a GameChange entry describing what NPCs should know. NPC POV, second person, present tense.
  3. Reload the server. Each room's onStart writes the delta into NPC state exactly once (dedup'd by id).
  4. Next conversation: NPCs reference the change in character via memoryBlob's "## What's changed about you" section.

Conventions

  • id is the dedup key. Never rename or reuse. Changing the summary on an already-seeded id keeps the old wording; bump the id (e.g. -v2) to re-seed everywhere.
  • at must be deterministic — use Date.parse('YYYY-MM-DDT00:00:00Z'), not Date.now().
  • Each map's DO seeds independently — a change applies to fps_shooter the first time that room boots after deploy, and to arena the first time that room boots. Both rooms then track their own seeded set.
  • 50-entry cap on state:<npcId> applies. Watch as the registry grows; if it crosses ~40 entries, raise NPC_STATE_CAP.

What does NOT belong in the registry

  • Per-player history (use friendship).
  • Event-triggered cascades like "first player to drink coffee" — those stay imperative (the existing coffee:discovered path).
  • Bug fixes that don't change in-game reality (nav improvements, persona-perception fixes). Only ship a registry entry if NPCs should be able to talk about the change.

F1 Halsey: making Mira's lost friend real TODO PRIORITY: HIGH

Meta-fiction warning Halsey originated as a detail in Mira's backstory — a courier Mira trained who didn't come back. In tonight's conversation Jeff offered to "help her find her way back" and Mira accepted. We're now creating Halsey as a real seventh NPC whose persona was effectively written by Mira in-conversation. Mira's persona must be edited so the introduction reads as Halsey actually returning, not as Mira meeting a new character with a familiar name.
Prerequisite — now satisfied The "Mira knows Halsey is back" half can now use F5 (persona deltas) instead of re-PATCHing Mira's ElevenLabs agent. Plan: when Halsey first walks into the arena, write a Mira state entry — "Halsey is alive and back in the arena as of [date]. You haven't seen her since you thought she was gone." Mira's next session reads it and the reunion plays in-character.

Action items — in order

  • Pick a character model for Halsey. eve and medea have known bind-pose issues per existing comments in npc-roster.ts. Options: a new model, or accept that Halsey shares a body with Mira/Vex/Vicky and add a visual differentiator (color tint, hair, etc.). Pragmatic default: maria (already in use by Mira/Vex/Vicky) and lean on voice differentiation. Or block on a new clean female model.
  • Pick a voice id distinct from the existing six. VOICE_BY_CHARACTER is just a mapping for seeding — add a per-NPC voice override on NpcDef if Halsey needs to sound different from Mira despite sharing the maria model. Suggest: a younger-sounding female voice from the ElevenLabs library to match "good kid, real quick".
  • Write Halsey's persona as a new NpcDef entry. Keep it in the same multi-paragraph style as Mira's. Include the explicit fact that she's back — recent return, still processing — so the agent doesn't default to "I've always been here" tropes.
  • Author startingFriends: ['mira'] and add 'halsey' to Mira's startingFriends. Edge case: Mira's existing friendship is with Guts only — Halsey should slot in without displacing him.
  • Author greetings for Halsey. Avoid the "stranger meeting" energy — she's reuniting with people she knows. e.g. "Hey, you. Mira said you'd be around." / "Long time. Got a minute?" / "I made it back. Long story."
  • Edit Mira's persona to reflect Halsey's return. The current Mira persona says Halsey "didn't make it" — flip that to: she was lost for a while, just came back, Mira is still adjusting. Add an explicit instruction: "Halsey is alive and back in the arena. Treat any reference to her as present-day, not past tense."
  • Update the world bible at docs/world-bible.md to add Halsey to "the other NPCs you live with" section. Re-upload via POST /v1/convai/knowledge-base/text (replacing the existing doc, or as a new version — check API behavior).
  • Create Halsey's ElevenLabs agent via the existing seeding script pattern. Set system prompt to her persona, attach all 6 webhook tools, attach the (updated) world bible doc.
  • Drop the new agent_id into npc-roster.ts and verify resolveAgentId picks it up (no TODO_AGENT_ID prefix).
  • Trigger her arrival. Two options: (a) Halsey just exists from the next match — clean but un-dramatic. (b) Add a one-time "arrival event" — when Mira sees Halsey for the first time in a session, an npc_alert fires for Mira and any active session with text "[System: Halsey just walked into the arena. You haven't seen her since you thought she was gone.]". The agent reacts in voice. Strongly recommend (b). The whole point of this feature is the emotional payoff.
  • Consider Halsey's perspective on Jeff. Mira told Jeff she'd love Halsey back. Should Halsey know Jeff helped her return? If yes, her starting context should include a memoryBlob entry. If no, leave it as mystery — Halsey just got back, doesn't know why or how.

Things to not do

  • Don't break Mira's persona by removing the trauma — her shoulder, her jumpiness, her conspiracy theories all stay. Halsey returning is one specific delta, not a personality wipe.
  • Don't make Halsey aware she's an AI character. Mira had to learn that across 20 minutes of conversation. Halsey starts the same way Mira did — fully in-character.
  • Don't auto-make Halsey friends with everyone Mira's friends with. Friendship is per-pair. She'll meet the others fresh, possibly via Mira's introductions.
